# Website Development Guide

This is the complete reference for working on the Westside website locally. Everything here runs on your desktop PC — no cloud services, no accounts, no subscriptions.

---

## The One Command You Need

```bash
cd C:/Projects/Westside-Website-v2
./dev.sh
```

This does three things:
1. Starts a **local web server** that hosts your website at `https://localhost:3000`
2. Opens a **normal Edge window** showing the desktop version of the site
3. Opens a **phone-sized Edge window** (390×844 pixels) showing how the site looks on mobile

**That's it.** Now edit any HTML, CSS, or JS file — every open browser window updates automatically. CSS changes appear instantly (no page reload). HTML and JS changes trigger a quick refresh.

Press **Ctrl+C** in the terminal to stop everything.

> You can also run `npm run dev` — it does the same thing.

## How It Works

### BrowserSync (the engine behind everything)

BrowserSync is a local web server designed for web development. When you run `./dev.sh`, it:

- **Serves your website files** over HTTPS, just like a real web server would
- **Watches for changes** — when you save a file, every connected browser updates
- **Syncs interactions** across all connected browsers — scroll on one, they all scroll; click a link on one, they all navigate

It prints two URLs when it starts:
- **Local:** `https://localhost:3000` — for browsers on this computer
- **External:** `https://100.98.144.48:3000` — for your phone or other devices (works over Tailscale from anywhere)

It also has a **dashboard** at `http://localhost:3001` where you can toggle sync features, see connected devices, and access other settings.

### Why HTTPS?

The local server uses HTTPS with trusted certificates (generated by **mkcert**). This matters because some modern web features (service workers, geolocation, some APIs) only work over HTTPS. More importantly, it means your local testing environment matches how the real site works — no surprises when you go live.

Chrome and Edge trust these certificates automatically. For Firefox, you'd need to go to `about:config` and set `security.enterprise_roots.enabled` to `true`.

### The two Edge windows

**Desktop window** — A normal Edge browser window. This is where you'll spend most of your time. Use Edge's DevTools (press **F12**) to inspect elements, debug CSS, check the console, etc.

**Mobile preview window** — A phone-sized window running in Edge's "app mode" (no tabs or address bar — the window *is* the viewport). It also sends a mobile user-agent string, so the site thinks it's being viewed on a Pixel phone. This triggers all your responsive CSS (`@media` queries) and gives you a quick visual check of the mobile layout.

This mobile window is great for quick visual feedback as you work. For more thorough mobile testing, see the next section.

## Mobile Testing Options

Listed from quickest to most thorough:

### 1. The phone-sized Edge window (automatic)

Already opens when you run `./dev.sh`. Good for: quick visual checks, spotting layout breaks, verifying responsive behavior as you code.

### 2. Edge DevTools device emulation (more accurate)

In any Edge window:
1. Press **F12** to open DevTools
2. Press **Ctrl+Shift+M** to toggle the device toolbar
3. Pick a device from the dropdown (iPhone 14, Pixel 7, Galaxy S20, etc.)

This is more accurate than a resized window because it also simulates:
- **Touch events** (instead of mouse clicks)
- **Device pixel ratio** (retina vs non-retina)
- **Network throttling** (3G, 4G, offline)

Good for: testing specific devices, debugging touch interactions, checking how the site feels on slower connections.

### 3. Firefox Responsive Design Mode (cross-browser check)

Open `https://localhost:3000` in Firefox, then press **Ctrl+Shift+M**. Firefox has its own rendering engine (Gecko vs Edge's Chromium), so things can look slightly different. Worth checking periodically.

### 4. Your actual phone

Open the External URL on your Pixel's browser:
```
https://100.98.144.48:3000
```
This works over Tailscale — your phone doesn't need to be on the same WiFi. You'll see the real site on a real phone, with real touch scrolling, real screen density, and real mobile browser behavior.

> **Note:** Your phone's browser may show a certificate warning since the mkcert certificate is only trusted on your PC. Tap "Advanced" → "Proceed" to bypass it.

For **Chrome remote debugging** (inspect your phone's browser from your desktop):
1. Connect your Pixel via USB
2. Enable USB Debugging in Developer Options on the phone
3. Run `adb reverse tcp:3000 tcp:3000` in the terminal
4. Open `https://localhost:3000` on your phone's Chrome
5. On your desktop, go to `chrome://inspect` or `edge://inspect` — you'll see your phone's tabs and can inspect them with full DevTools

## Other Tools

These are all available as `npm run ___` commands. You don't need to use them right away — they're there when you need them.

### Code Quality

| Command | What it does |
|---------|-------------|
| `npm run lint:html` | Checks all HTML files for errors (missing tags, invalid attributes, accessibility issues) |
| `npm run lint:css` | Checks CSS for errors and bad practices |
| `npm run lint:js` | Checks JavaScript for bugs and bad patterns |
| `npm run lint` | Runs all three linters |
| `npm run format` | Auto-formats all HTML, CSS, and JS files (consistent indentation, spacing, quotes) |
| `npm run format:check` | Shows what *would* change without actually changing anything |

### Site Audits

These test the live site (BrowserSync must be running first):

| Command | What it does |
|---------|-------------|
| `npm run audit:a11y` | Tests accessibility (WCAG 2.0 AA) — can people with disabilities use the site? |
| `npm run audit:links` | Finds broken links across the entire site |
| `npm run audit:lighthouse` | Full Google Lighthouse audit (performance, SEO, accessibility, best practices) — saves an HTML report |

### Sharing

| Command | What it does |
|---------|-------------|
| `npm run tunnel` | Creates a temporary public URL (like `https://abc123.trycloudflare.com`) that anyone can access — great for showing the site to someone on their phone or another computer without any setup |

## What Each Tool Is

A quick reference for everything that was installed:

| Tool | What it is | When you'd use it |
|------|-----------|------------------|
| **BrowserSync** | Live-reloading dev server | Every time you work on the site — it's the foundation |
| **serve** | Simpler static file server (no live reload) | Quick one-off checks; `serve .` and done |
| **mkcert** | Generates trusted local HTTPS certificates | Already set up — you shouldn't need to touch it |
| **cloudflared** | Creates instant public tunnels to your local server | Sharing the site with someone who isn't on your network |
| **ESLint** | JavaScript linter | Finding bugs and bad patterns in `main.js` |
| **Stylelint** | CSS linter | Finding errors in stylesheets |
| **Prettier** | Code formatter | Making all your code consistently formatted |
| **html-validate** | HTML validator | Catching invalid HTML before it causes browser quirks |
| **pa11y** | Accessibility tester | Making sure the site works for everyone |
| **Lighthouse** | Comprehensive site auditor | Performance, SEO, accessibility — the full picture |
| **linkinator** | Broken link checker | Finding dead links before visitors do |
| **webhint** | Broad best-practices checker | Security headers, compatibility, performance hints |

You also already had **Playwright** (automated browser testing) and **Lighthouse** (installed earlier) — those still work the same way.

## Tips

- **You don't need to restart BrowserSync** when you change HTML/CSS/JS — it picks up changes automatically. You only need to restart if you change `bs-config.js` itself.
- **Multiple browsers at once** — Open `https://localhost:3000` in as many browsers as you want (Edge, Firefox, Chrome). They all get live reload. This is great for cross-browser testing.
- **The BrowserSync UI** at `http://localhost:3001` lets you turn off scroll/click syncing if it gets annoying (it can be distracting when you're trying to look at different parts of the page in different windows).
- **Ctrl+C** cleanly stops everything. If BrowserSync ever gets stuck, close the terminal window.
